# ClaudeCoach 后端数据储备说明

## 1. 先说明真实架构

ClaudeCoach 没有独立部署的远程后端服务。

它的“后端”本质上是：

- VS Code 扩展宿主进程
- 一套本地文件系统存储
- 若干负责课程、资料、偏好、诊断、AI 配置的 TypeScript 模块

所以这里说的“后端数据储备”，准确表达应该是：

```text
本地持久化层 + 扩展宿主侧的数据管理层
```

## 2. 数据根目录设计

数据根目录由 `src/storage/pathResolver.ts` 统一计算。

默认根目录：

```text
~/ClaudeCoach
```

如果用户配置了 `claudeCoach.dataDirectory`，则优先使用配置值。

此外，项目还区分：

- 全局级数据
- 工作区级数据
- 资料库数据
- 旧版兼容数据

## 3. 目录树结构

当前实际目录结构可以概括为：

```text
ClaudeCoach/
  app/
    ai/
      profiles.json
    user/
      profile.json
    preferences/
      learning.json
    diagnostics/
      latest.json
      diagnosis-report.md
      history/
      <subject>/
        latest.json
        history/
        <subject>-diagnosis-report.md

  library/
    materials/
      index.json
      <subject>/
        <materialId>/
          source.pdf|txt|md
          extracted.txt
          summary.json
          meta.json

  workspaces/
    <workspaceId>/
      meta.json
      ai/
        override.json
      courses/
        <subject>/
          outline.json
          summary.md
          topics/
            <topicId>/
              summary.json
              lessons/
                <lessonId>.md
              exercises/
                <lessonId>/
                  prompt.md
                  prompt.json
                  grade.json
                  feedback.md
```

这套结构有两个很重要的设计点：

- 同一台机器上的不同 VS Code 工作区彼此隔离
- 资料库是全局库，课程是工作区库

## 4. 模块职责拆解

### 4.1 `StoragePathResolver`

职责：

- 统一生成所有文件路径
- 处理默认目录与自定义目录
- 生成工作区唯一 ID
- 兼容旧版 `courses/` 目录

模块接口语言：

```text
模块名：StoragePathResolver
输入：subject / topicId / lessonId / materialId / workspace
处理：拼接标准路径、清洗路径段、生成目录层级
输出：所有持久化文件的绝对路径
```

### 4.2 `CourseManager`

职责：

- 读取 / 保存课程大纲
- 规范化主题编号与课时编号
- 创建课程目录结构
- 管理讲义路径和练习路径
- 维护课时状态
- 迁移旧版课程文件
- 汇总 topic 级统计

它是课程数据层的核心入口。

### 4.3 `MaterialManager`

职责：

- 导入资料
- 维护资料总索引 `index.json`
- 保存源文件、副本、提取文本、摘要 JSON、meta JSON
- 为 AI 提供资料摘要与检索片段

### 4.4 `ProgressStore`

职责：

- 保存学生画像 `profile.json`
- 记录总学习次数
- 记录总练习次数

### 4.5 `PreferencesStore`

职责：

- 保存学习偏好 `learning.json`
- 读取难度、节奏、语言偏好

### 4.6 `AdaptiveEngine`

职责：

- 汇总 topic summary 与 grade history
- 生成学习诊断结果
- 保存 `latest.json`
- 存档历史诊断
- 生成 Markdown 诊断报告

## 5. 课程数据如何落盘

### 5.1 课程大纲

生成课程后，系统至少会写入：

- `outline.json`
- `summary.md`

其中：

- `outline.json` 是机器可读结构
- `summary.md` 是方便人阅读的课程摘要

伪代码：

```pseudo
saveCourseOutline(subject, outline):
  normalized = normalizeOutline(subject, outline)
  ensureDir(courseSubjectDir)
  ensureDir(courseTopicsDir)
  writeJson(outlinePath, normalized)

  for topic in normalized.topics:
    ensureDir(topicDir)
    ensureDir(topicLessonsDir)
    ensureDir(topicExercisesDir)
```

### 5.2 主题与课时编号规范化

系统不会直接相信 AI 给出的 `topic-01`、`lesson-01`。

它会重新生成更稳定的编号：

- 主题：`01-chapter-xxx`
- 课时：`01-03-xxx`

这一步的作用是：

- 让文件名稳定
- 保持排序自然
- 便于后续用消息或聊天定位讲义文件

### 5.3 讲义文件

每节讲义保存为：

```text
lessons/<lessonId>.md
```

讲义是 Markdown 文件，不是数据库记录。这样做的好处是：

- 用户可直接在编辑器里看和改
- AI 也能直接对 Markdown 文件做修改回写

### 5.4 练习文件

每节练习会同时落两类文件：

- `prompt.md`：给人做题
- `prompt.json`：给程序和批改器读取

伪代码：

```pseudo
generateExercises():
  exercises = ai.generateJson()
  writeMarkdown(prompt.md)
  writeJson(prompt.json)
```

## 6. 课时状态如何计算

课时状态不是单独靠一个数据库字段维护，而是“文件状态 + 内容解析”的结果。

状态计算规则大致如下：

- 没有讲义、没有练习：`not-started`
- 有讲义或有练习：`in-progress`
- 练习文件中每题都有答案且都插入了批改结果：`completed`

伪代码：

```pseudo
resolveLessonStatus():
  if exerciseFile exists:
    parse markdown sections
    if every section has answer and alreadyGraded:
      return completed

  if lessonFile exists or exerciseFile exists:
    return in-progress

  return not-started
```

这说明系统把“学习进度”绑定到了真实学习产物上，而不是只绑在按钮点击事件上。

## 7. 资料库数据如何落盘

### 7.1 资料导入

导入资料后，系统会：

1. 把原始文件复制到资料库目录
2. 生成 `meta.json`
3. 把资料登记到 `index.json`
4. 异步执行文本提取与摘要构建

伪代码：

```pseudo
importMaterial(subject):
  sourceFile = userSelectFile()
  id = "mat-" + timestamp
  storageDir = materialDir(subject, id)

  copy sourceFile -> storageDir/source.ext
  write meta.json
  append entry -> index.json

  async processEntry(entry)
```

### 7.2 资料处理状态

资料处理分三阶段：

- `pending`：刚导入，还没处理
- `extracted`：文本已提取
- `indexed`：摘要已生成，可以参与 AI 检索

### 7.3 资料派生文件

每份资料会形成下面几类派生数据：

- `source.xxx`：原文件副本
- `extracted.txt`：抽取后的纯文本
- `summary.json`：结构化章节摘要
- `meta.json`：资料元信息

这套设计本质上是在做一个轻量的“本地知识底座”。

## 8. 学习偏好与学生画像

### 8.1 学生画像

`profile.json` 保存：

- 姓名
- 水平
- 学科列表
- 目标
- 开始时间
- 总学习次数
- 总练习数

### 8.2 学习偏好

`learning.json` 保存：

- 全局难度
- 各学科难度覆盖
- 练习难度配比
- 每日目标分钟数
- 每次练习数量
- 学习速度
- 复习频率
- 内容语言
- 代码注释语言

这些数据不是“展示型装饰”，而是会进入 Prompt 上下文。

## 9. 批改与诊断数据

### 9.1 单次批改数据

每次批改至少会产生：

- `grade.json`
- `feedback.md`

`grade.json` 面向机器读取：

- 分数
- feedback
- strengths
- weaknesses
- gradedAt

`feedback.md` 面向人阅读。

### 9.2 Topic 级汇总数据

每个主题会维护一个 `summary.json`，里面记录：

- 累计练习次数
- 平均分
- 历史分数列表
- 错误类型计数
- 最后更新时间

这是学习诊断的原始输入之一。

### 9.3 学习诊断数据

诊断结果会写入：

- `latest.json`
- `history/<date>.json`
- `diagnosis-report.md`

工作方式：

- 永远保留当前最新版本
- 上一次结果会归档
- 同时生成可读 Markdown 报告

## 10. 旧版数据迁移

当前数据层已经实现了旧版兼容迁移，意味着新旧目录都可能存在。

迁移逻辑包括：

- 旧版课程大纲迁移到新 `outline.json`
- 旧版课程摘要迁移到新 `summary.md`
- 旧版讲义迁移到新 `lessons/`
- 旧版练习与批改文件迁移到新 `exercises/<lessonId>/`
- 旧版 AI profiles / profile / preferences / materials 迁移到新路径

所以这个项目的数据层不是从零开始的纯新结构，而是“新结构 + 兼容旧结构”的演进式设计。

## 11. 删除与重置行为

这里要注意两个行为边界。

### 11.1 删除课程

当前“删除课程”只会移除课程大纲文件，不会删除讲义和练习实体文件。

也就是说：

```text
这是“从课程列表移除”，不是“彻底清空课程所有产物”。
```

### 11.2 重置课时

重置课时会删除：

- 讲义文件
- 对应练习目录

然后把状态设回 `not-started`。

## 12. 后端数据层伪代码总览

```pseudo
UserAction
  -> SidebarProvider
  -> CourseManager / MaterialManager / ProgressStore / PreferencesStore / AdaptiveEngine
  -> writeJson / writeText / writeMarkdown
  -> local filesystem
  -> SidebarProvider aggregates result
  -> Webview rerender
```

更细一点：

```pseudo
生成课程:
  AI -> outline data
  CourseManager.normalizeOutline()
  write outline.json
  write summary.md

导入资料:
  copy source
  write meta.json
  update index.json
  extract text
  write extracted.txt
  AI summarize
  write summary.json

生成练习:
  AI -> exercises json
  write prompt.md
  write prompt.json

批改:
  AI -> grade result
  write grade.json
  write feedback.md
  update topic summary

诊断:
  collect topic summaries + grades
  AI -> diagnosis json
  archive old diagnosis
  write latest.json
  write diagnosis-report.md
```

## 13. 一句话总结

ClaudeCoach 的“后端数据储备”不是数据库服务器，而是一套围绕课程、资料、偏好、批改、诊断和 AI 配置构建的本地文件仓库。它的价值不在“高并发服务”，而在“把学习过程中的每个产物都沉淀成可追踪、可重用、可再次喂给 AI 的本地资产”。
